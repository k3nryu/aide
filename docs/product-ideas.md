# 产品想法记录

这份文档先记录当前产品方向想法，不急着变成完整需求。后续可以再拆成数据模型、API、UI、路线图。

## Task

Task 不应该只是简单的 todo list，而是一个个人执行系统。它需要清楚区分 To-Do 和 Not-to-Do，支持非定期任务、定期任务、计划与复盘、自动安排，以及和 not-to-do 的冲突提醒。

### 任务类型

任务先分为两个顶层类型：

- To-Do：需要完成或处理的事情。
- Not-to-Do：持续有效的边界和提醒，不应该有完成概念。

To-Do 再分为：

- 非定期任务：手动添加，或未来从 Google Calendar、Apple Calendar、Outlook Calendar 等日历自动获取。
- 定期任务：每天、每周、每月、每年的固定或周期性事项。

任务还应该支持场景分类：

- 公司
- 个人

这个分类用于过滤、查看和自动安排。例如工作日白天可以优先显示公司相关任务，晚上和周末可以更自然地显示个人、家庭、健康和学习相关任务。

当前原型已支持在任务上保存公司/个人分类，并在任务一览里按分类过滤。

定期任务应该支持国历和农历。初始表达不追求复杂规则引擎，先保存结构化频率和自然语言描述。

定期任务示例：

- 每天学习 X 个单词。
- 每周四、周六上游泳课。
- 每半年更新一次定期券。
- 每个月检查一次工资卡。
- 每年农历某月某日的生日，比如自己的生日、老婆的生日。准备时间和紧急程度由 Aide 根据上下文评估，不要求用户填写“提前几天”。

每日简报应该能把当天需要关注的任务列出来，包括由定期任务自动生成或推导出来的今日事项。

### 日历和会议

Aide 未来应该可以从外部日历自动提取当天会议，并把它们纳入今日简报和任务视图。

优先考虑的日历来源：

- Google Calendar
- Apple Calendar
- Outlook Calendar，特别是公司日历

会议不一定都应该变成普通任务。更合理的方式是把会议作为当天上下文，必要时自动生成准备事项、会后 follow-up、资料整理或提醒。

当前原型已支持统一的会议数据模型、今日会议录入接口和今日简报展示。Google Calendar、Apple Calendar、Outlook Calendar 的真实自动同步需要后续接入账户授权、provider API 或 CalDAV/ICS 同步。

### 定期任务规则

任务添加功能默认是非定期 To-Do，不需要用户选择 todo/not-to-do 类型。用户可以勾选“这是定期任务”，再填写定期任务信息。

定期任务输入方式：

- 默认用自然语言输入，例如“每周四和周六游泳”“每半年更新定期券”“每年农历生日”。
- 同时提供简单结构化字段：每天、每周、每月、每年，国历或农历，月、日、星期。
- 不提供高级规则输入，不要求用户写 cron。
- 不提供 Markdown/YAML 高级输入。复杂解析以后交给 Aide 或 AI 服务处理。

定期任务大多代表用户长期想坚持、维护或提前准备的事情，默认可以偏重要。但紧急程度不应该固定，也不要求用户填写“提前几天变紧急”，而应该根据频率、截止日期、自然语言描述和影响自动评估。

示例规则：

- 老婆生日一年一次，但需要提前准备。Aide 应根据生日、关系和准备成本自动提高提醒强度。
- 定期券半年更新一次，临近更新窗口时应进入需要注意状态。
- 每天学单词属于重要但不一定紧急，除非当天快结束还没完成。
- 每周固定课程在当天或前一天进入更高紧急度，平时保持为重要但不紧急。

这种规则的目标不是制造压力，而是让 Aide 在正确的时间把事情推到用户眼前。

当前原型已支持保存非定期/定期任务、自然语言定期描述、频率、国历/农历、月、日、星期等元数据。规则解析、农历日期计算、未来实例自动生成和更聪明的 AI 评估先保留为下一步能力。

### 每日简报展示

每日简报是首页的核心内容。

需求：

- 今日简报和状态概览放在第一页。
- 今日简报里的每一条报告都应该可以打开查看详情。
- 报告详情尽量用 5W2H 格式呈现：
  - What：是什么事情。
  - Why：为什么重要，或为什么需要注意。
  - Who：相关的人。
  - When：时间、截止日期或发生时间。
  - Where：地点。
  - How：执行方式、处理方式或注意事项。
  - How much：金额、数量、频率、成本或投入。
- 每条简报应该保持简短，但可以展开看到更结构化的信息。

### PDCA

任务需要支持 PDCA：

- Plan：计划要做什么。
- Do：执行任务。
- Check：检查结果和执行情况。
- Act：调整任务、习惯或后续计划。

这可以让 Aide 不只是记录任务，还能帮助用户持续改进执行方式。

### 四象限分类

任务需要支持重要/紧急四象限：

- 重要且紧急
- 重要但不紧急
- 不重要但紧急
- 不重要且不紧急

Aide 未来应该根据任务的重要程度、紧急程度、截止日期、定期规则和用户状态，自动帮助安排任务。

优先级不应该主要依赖用户手动输入。Aide 应根据重要度和紧急度自动评估优先级。

初始规则：

- 重要但不紧急：超高优先级。长期成长、健康、学习、系统建设等事情通常属于这一类，应该主动安排。
- 重要且紧急：高优先级。必须处理，但如果长期都集中在这一类，说明系统过于被动。
- 不重要但紧急：中优先级。需要控制投入，避免被紧急感牵着走。
- 不重要且不紧急：低优先级。尽量减少或不做。

产品倾向：鼓励用户优先处理“重要但不紧急”的事情，因为这类事情最容易被拖延，但最能改善长期状态。

### 任务页面体验

任务页面应该降低输入阻力。

需求：

- 页面切换顺序优先为：今日、任务、财务、想法、行为。
- 手机端导航放在底部，方便点击。
- 电脑端导航放在顶部或侧边，方便点击并减少操作。
- 任务添加默认只显示任务输入框。
- 用户选中或展开后，才显示其他输入项，比如描述、分类、重要度、截止日期和定期任务设置。
- 新增任务默认就是 To-Do，不需要选择 todo/not-to-do 类型。
- 定期任务通过一个开关进入，不需要高级规则、高级输入或“提前几天变紧急”字段。
- 任务优先级由 Aide 根据重要度和紧急度自动评估，不需要用户手动输入。
- 新增任务的紧急度默认由 Aide 根据截止日期和定期属性评估；任务一览编辑里可以保留紧急度修正入口，方便用户纠正 Aide 的判断。
- 任务一览中只显示 todo 类任务。
- 任务一览中的任务可以展开编辑，至少支持修改标题、描述、重要度、紧急度和截止日期。
- 优先级高的任务排在前面，同等情况下优先显示重要度、紧急度更高或截止日期更近的任务。
- 如果截止日期已经接近，Aide 应自动提高该任务的紧急度，避免临近任务被忽略。
- Not-to-do 不直接混在任务一览中，而是通过打开/折叠的方式查看。
- 任务一览需要显示任务的重要度和紧急度。
- 任务一览需要支持按“公司/个人”分类查看。
- 添加任务后，任务应该立即显示在任务一览中。

### 已完成任务

完成的任务应该可以在“已完成”里查看，但不要默认展示全部历史，避免页面变成档案库。

查看方式：

- 最近 1 天
- 最近 3 天
- 最近 1 周
- 最近 1 个月
- 最近 X 年
- 按关键词搜索

这个功能既可以用于回顾执行情况，也可以帮助用户找回以前做过的事情。

当前原型已支持在浏览器 UI 中按时间范围和关键词查看已完成任务。

### 大任务和 WBS

大的任务应该可以做 WBS 拆分。

未来 UI 可以把大任务可视化为树状结构、清单、时间线或其他适合执行的结构，而不是只显示成一个模糊的大 todo。

### 和 Not-To-Do 的冲突提醒

如果 task 违背了 not-to-do，Aide 应该给出警告。

例如，一个任务如果违反法律边界、道德良心、公司规则、家庭原则或身体健康原则，系统应该在计划或执行前提醒用户。

## 行为记录

除了 task，Aide 还应该支持记录用户实际做过的事情。

这些事情不一定提前出现在 task list 里，也不一定需要计划，但它们对长期分析很有价值。

示例：

- X 点上了一次厕所。
- X 日去了一趟 X 地。
- X 点吃了某种食物。
- X 点睡觉或起床。
- X 日运动了一次。
- X 日出现了某种身体不舒服。
- X 日做了某个临时决定或临时行动。

这些记录更接近 life log 或 activity log，不应该强行当成 todo。

未来用途：

- 推测健康程度。
- 分析行为趋势。
- 发现不良习惯。
- 发现和强化良好习惯。
- 辅助生成每日、每周、每月复盘。
- 和 task、not-to-do、财务记录、AI memory 产生关联。

行为记录需要尽量低摩擦，最好可以快速输入一条自然语言记录，之后再由系统自动分类、打标签或生成结构化数据。

行为记录输入默认只显示必须填写的内容，例如“做了什么”。分类、时间、备注等字段应该收进详细设置里，用户需要时再展开。

## Not-To-Do

Not-to-do 应该是一等公民，不只是普通备注。它代表用户明确不想做、不能做、或不应该做的事情。

初始分类：

- 违法的事情不做。
- 违背道德良心的事情不做。
- 违背公司规则的事情不做。
- 对家人不好的事情不做。
- 对自己身体不好的事情不做。

用户应该可以随时在这些分类下添加新的 not-to-do。

这些规则未来应该可以用于检查 task，并在发现冲突时给出提醒。

Not-to-do 不应该有“完成”概念，因为它代表持续有效的边界，而不是一次性任务。

Not-to-do 的输入输出应该保持简单：

- 输入：一句边界内容，加一个分组。
- 分组：法律、道德、公司、家庭、健康。
- 输出：按分组折叠/展开查看，不混入普通任务一览。
- 暂时不提供高级输入、Markdown 或 YAML。复杂原则、例外条件和外部规则以后可以由 Aide 逐步结构化。

## 财务

财务功能的长期方向可以接近 Money Forward。

想法包括：

- 通过各个银行的 API 获取或同步数据。
- 可视化当前财务状态。
- 记录收入、支出、账户、资产等信息。
- 记录财富积累目标。
- 计算距离实现财富自由还差多少钱。

这个方向会涉及隐私、安全、银行 API、数据导入和可视化，属于比较大的功能模块。

### 财务页面体验

当前阶段财务页面不需要很细的流水记录。

更重要的是作为“财富自由仪表盘”，快速回答：

- 现在有多少真正参与财富自由计算的 FI 资产。
- 每月家庭固定支出项目合计是多少。
- 每月被动收入是多少。
- 被动收入覆盖率是多少。
- 按安全提取率反推，FI 目标本金是多少钱。
- 距离实现财富自由还差多少钱。
- 按当前每月新增投资，还需要多久。

流水、分类、银行 API 和复杂可视化可以放到后面。当前 UI 应优先展示财务自由相关的关键数字。

财富自由定义：

- 核心公式：被动收入 >= 生活总支出。
- 本质：不再为了生活开销被迫出售时间，获得时间自由和时间支配权。
- 真正参与 FI 计算的资产应该是能产生现金流、投资收益或未来可变现投资收益的资产。
- 自住房、车、难变现物品等可以记录为非 FI 资产，但默认不计入 FI 目标计算。

当前计算规则：

现金流法：

- 被动收入覆盖率 = 每月被动收入 / 每月生活支出。
- 每月生活支出由固定支出项目加总，例如房租、游泳、饮食、水电网等。
- 每月现金流差额 = 每月被动收入 - 每月固定支出，可以是负数。
- 如果每月被动收入已经覆盖每月生活支出，FI 差距可以视为 0。

本金法：

- FI 目标本金 = 年固定支出 / 安全提取率。
- FI 差距 = FI 目标本金 - 当前 FI 资产。
- 每月新增投资不改变 FI 目标本金或 FI 差距本身，但用于估算补齐 FI 差距还需要多久。
- 安全提取率需要可调，默认使用 3.5%，也可以按 3%、4% 等假设查看。

当前原型已将财务页改为 FI 仪表盘，支持 FI 资产、非 FI 资产、固定支出项目增删、每月被动收入、每月新增投资和安全提取率。UI 应将现金流法和本金法分组展示，并用短文字说明每个数字代表什么。

### 多币种资产和汇率

资产输入需要支持多种资产类型：

- 日元
- 人民币
- 美金
- 虚拟货币，例如 BTC、ETH

显示需求：

- 最终结果可以用日元或人民币展示。
- 财务页面需要有日元/人民币切换按钮。
- 汇率根据当天汇率自动计算。
- 资产金额必须标明单位。
- FI资产、FI目标本金、还差金额、每月支出、每月被动收入、现金流差额、非FI资产都需要同时显示原始大数字和括号里的小字概算。
- 小字概算用更容易理解的单位，例如“约 X 万”、“约 X 千万”、“约 X 亿”。
- 括号里的金额只放金额概算，不混入提取率、状态说明等其他文本；其他说明放在单独小字里。
- 距离财务自由还差的时间，需要根据每月新增投资自动换算成“X年X个月X天”。
- 每月新增投资额需要在本金法里显示，并说明它只影响“还需要多久”，不改变 FI 目标本金或本金缺口本身。
- 金额输入框应尽量像银行 App 一样，只接受数字输入，并自动添加千分位逗号。

当前阶段可以先做本地浏览器保存和前端换算。后续再考虑把财务概况持久化到后端数据库。

## Thoughts

用户已经有 Joplin 笔记本，可以随时记录想法。

所以 Aide 里的 thoughts 功能目前可以不是核心优先级。未来可以考虑：

- 保留一个轻量 capture 功能。
- 以后和 Joplin 集成。
- 从 Joplin 中导入部分笔记进入 Aide memory。
- 只把需要 follow-up、转成任务、或进入 AI memory 的内容放进 Aide。

想法页面应作为独立页面，不和任务、行为记录混在一起。

想法输入也应该低摩擦。默认只显示必须填写的内容，例如“内容”。标签等字段放进详细设置里。

## 公网访问和认证

用户计划通过 Cloudflare Tunnel 把 Aide 放到公网访问。

认证需求：

- 安全性要比较高。
- 操作要简单。
- 尽量不需要记住密码。
- 因为用户主要只有自己，所以访问控制可以围绕自己的邮箱、手机、电脑和可信设备设计。

倾向方案：

- 使用 Cloudflare Access 保护 Aide。
- 使用邮箱一次性验证码、Google 登录、Passkey 或设备策略，而不是先自己实现用户名密码。
- 只允许用户自己的邮箱或可信设备访问。
- 后端未来可以校验 Cloudflare Access 注入的身份信息或 JWT，作为额外保护。

## 当前优先级倾向

根据这些想法，当前产品优先级可以倾向于：

1. 强化 task 系统：定期任务、每日简报、PDCA、四象限、WBS。
2. 增加行为记录：记录实际发生过的事情，用于健康、习惯和趋势分析。
3. 优化移动端和桌面端 UI：减少操作，首页聚焦今日简报和状态概览。
4. 强化 not-to-do：分类规则、随时添加、task 冲突提醒。
5. 扩展财务：先展示 FI 资产、每月支出、被动收入覆盖率、FI 目标本金和差距，再考虑复杂流水。
6. 做好公网访问认证：优先使用 Cloudflare Access，不急着自研密码系统。
7. 降低 thoughts 优先级：先作为独立可选页面，未来考虑 Joplin 集成或 AI memory 输入源。
