# 产品想法记录

这份文档先记录当前产品方向想法，不急着变成完整需求。后续可以再拆成数据模型、API、UI、路线图。

## 个人成长 PDCA 助手

Aide 的核心不是复杂日程系统，而是一个持续循环：

- 用户行动。
- 用户用一句话或一次对话记录发生了什么。
- Aide 帮忙保存行动轨迹、检查偏差、总结模式。
- Aide 给出下一步最小动作、任务草稿或想法输入。
- 用户继续行动，再反馈。

行为记录应该支持极简输入，也允许按 PDCA/SOP 方式补充结构化信息：Plan、Result、Check/Learning、Act/Next action、能量状态。SCAQ、SMART、AIDA、5W2H 等模型应该作为沟通和复盘格式存在，帮助用户快速聚焦，不应该变成复杂表单。

当前原型的行为复盘已经可以生成 PDCA、STOW、SCAQ、SMART、AIDA、5W2H 和辩论问题草稿。这些输出先由本地规则生成，未来再替换成 AI provider，但用户确认和编辑的流程应该保持不变。

### 辩论式成长日志

Aide 和用户的日常聊天可以采用辩论方式：用户先记录行动、感受、想法或结果，Aide 再从证据、反例、长期目标和下一步行动四个角度追问。目标不是证明用户错了，而是帮助用户发现假设、减少自我欺骗，并把讨论落到一个可执行动作。

适合追问的问题：

- 这件事有什么证据证明它真的推进了长期自由？
- 如果反过来看，最强的反对理由是什么？
- 这是重要但不紧急的建设，还是只是临时忙碌？
- 如果今天只保留一个 15 分钟动作，应该保留哪一个？

### 日程边界

Aide 不应该重新实现复杂日程系统。工作日、节假日、会议和提醒优先通过 CalDAV、Google Calendar、Apple Calendar、Outlook 或其他外部服务接入。Aide 的职责是读取这些上下文，放进今日简报和复盘闭环里，而不是成为另一个重型日历产品。

## Task

Task 不应该只是简单的 todo list，而是一个个人执行系统。它需要清楚区分 To-Do 和 Not-to-Do，支持非定期任务、定期任务、计划与复盘、自动安排，以及和 not-to-do 的冲突提醒。

### 任务类型

任务先分为两个顶层类型：

- To-Do：需要完成或处理的事情。
- Not-to-Do：持续有效的边界和提醒，不应该有完成概念。

To-Do 再分为：

- 非定期任务：手动添加，或未来从 Google Calendar、Apple Calendar、Outlook Calendar 等日历自动获取。
- 定期任务：每天、每周、每月、每年的固定或周期性事项。

任务还应该支持场景分类：

- 公司
- 个人

这个分类用于过滤、查看和自动安排。例如工作日白天可以优先显示公司相关任务，晚上和周末可以更自然地显示个人、家庭、健康和学习相关任务。

当前原型已支持在任务上保存公司/个人分类，并在任务一览里按分类过滤。

定期任务应该支持国历和农历。初始表达不追求复杂规则引擎，先保存结构化频率和自然语言描述。

定期任务示例：

- 每天学习 X 个单词。
- 每周四、周六上游泳课。
- 每半年更新一次定期券。
- 每个月检查一次工资卡。
- 每年农历某月某日的生日，比如自己的生日、老婆的生日。准备时间和紧急程度由 Aide 根据上下文评估，不要求用户填写“提前几天”。

每日简报应该能把当天需要关注的任务列出来，包括由定期任务自动生成或推导出来的今日事项。

### 日历和会议

Aide 未来应该可以从外部日历自动提取当天会议，并把它们纳入今日简报和任务视图。

优先考虑的日历来源：

- Google Calendar
- Apple Calendar
- Outlook Calendar，特别是公司日历

会议不一定都应该变成普通任务。更合理的方式是把会议作为当天上下文，归并到任务一览的执行流里展示；必要时再自动生成准备事项、会后 follow-up、资料整理或提醒。

会议也应该支持和任务类似的基础属性：

- 个人 / 公司分类。
- 重要度，默认重要。
- 非定期会议和定期会议。
- 定期会议支持每天、每周、每月、每年的简单周期字段。

当前原型已支持统一的会议数据模型、今日会议录入接口和今日简报展示。Google Calendar、Apple Calendar、Outlook Calendar 的真实自动同步需要后续接入账户授权、provider API 或 CalDAV/ICS 同步。

### 定期任务规则

任务添加功能默认是非定期 To-Do，不需要用户选择 todo/not-to-do 类型。用户可以勾选“这是定期任务”，再填写定期任务信息。

定期任务输入方式：

- 默认用自然语言输入，例如“每周四和周六游泳”“每半年更新定期券”“每年农历生日”。
- 同时提供简单结构化字段：每天、每周、每月、每年，国历或农历，月、日、星期。
- 不提供高级规则输入，不要求用户写 cron。
- 不提供 Markdown/YAML 高级输入。复杂解析以后交给 Aide 或 AI 服务处理。

定期任务大多代表用户长期想坚持、维护或提前准备的事情，默认可以偏重要。但紧急程度不应该固定，也不要求用户填写“提前几天变紧急”，而应该根据频率、截止日期、自然语言描述和影响自动评估。

示例规则：

- 老婆生日一年一次，但需要提前准备。Aide 应根据生日、关系和准备成本自动提高提醒强度。
- 定期券半年更新一次，临近更新窗口时应进入需要注意状态。
- 每天学单词属于重要但不一定紧急，除非当天快结束还没完成。
- 每周固定课程在当天或前一天进入更高紧急度，平时保持为重要但不紧急。

这种规则的目标不是制造压力，而是让 Aide 在正确的时间把事情推到用户眼前。

当前原型已支持保存非定期/定期任务、自然语言定期描述、频率、国历/农历、月、日、星期等元数据。规则解析、农历日期计算、未来实例自动生成和更聪明的 AI 评估先保留为下一步能力。

### 每日简报展示

每日简报是首页的核心内容。

需求：

- 今日页第一块应叫“今日 To-Do”，只放今天真正需要处理或参加的任务/会议。
- 今日 To-Do 里的每一条报告都应该可以打开查看详情。
- 今日 To-Do 不显示 Not-to-Do，也不显示固定的“Today's briefing from Aide.”占位卡片。
- 今日 To-Do 中已经完成/已经参加的任务和会议仍然显示，但放在底部并用删除线区分，避免刷新后不知道刚刚做完了什么。
- 今日 To-Do 底部的“已完成 / 已参加”默认折叠，点击后再展开查看，避免压住未完成事项。
- 今日页第二块应叫“今后 To-Do”，用于展示明天、这周、这个月、今年、明年、3年目标、5年目标。
- 今后 To-Do 的“明天预定”不显示非定期会议，避免把明天的普通会议提前挤到今天的注意力里。
- 今日页第三块应叫“今日简报”，只保留高层摘要。
- 今日简报显示今日未完成 To-Do 数量。
- 今日简报显示财务自由时间估算：按照当前每月固定支出和每月新增投资额估算，还需要多久实现财务自由。
- 今日简报不再显示 Not-to-Do 数量和行为记录数量。
- 今日 To-Do 只显示今天真正需要处理或参加的 To-Do/会议，不显示 Not-to-Do。
- 报告详情尽量用 5W2H 格式呈现：
  - What：是什么事情。
  - Why：为什么重要，或为什么需要注意。
  - Who：相关的人。
  - When：时间、截止日期或发生时间。
  - Where：地点。
  - How：执行方式、处理方式或注意事项。
  - How much：金额、数量、频率、成本或投入。
- 每条简报应该保持简短，但可以展开看到更结构化的信息。

首页还需要一个“预定”模块，用来把不是今天、但未来需要意识到的事项从今日简报里分离出来。

预定模块的分组：

- 明天预定。
- 这周预定。
- 这个月预定。
- 今年预定。
- 明年预定。
- 3年目标。
- 5年目标。

预定模块可以展示非定期任务、定期任务和会议的下次发生日期。今日简报负责“今天做什么”，预定模块负责“接下来有什么”，避免每周课程、生日、定期券这类未来事项挤占今日注意力。

今日简报里的任务和会议需要支持直接记录执行结果：

- 任务可以记录“完成 / 未完成”，会议可以记录“参加 / 未参加”。
- 未完成或未参加时，必须填写进度或理由，避免只留下一个空状态。
- 结果写入行为记录，用于后续 PDCA、STOW、健康和习惯分析。
- 定期任务和定期会议不应该因为一次完成就把本体标成永久完成；应按发生周期记录，例如“今日完成 / 今日参加”、“这周完成 / 这周参加”、“本月完成 / 本月参加”、“今年完成 / 今年参加”。
- 非定期任务完成时，可以同时写入行为记录并把任务本体标为已完成。

### PDCA

任务需要支持 PDCA：

- Plan：计划要做什么。
- Do：执行任务。
- Check：检查结果和执行情况。
- Act：调整任务、习惯或后续计划。

这可以让 Aide 不只是记录任务，还能帮助用户持续改进执行方式。

Thought、Task、行为记录之间应该形成 PDCA 闭环：

- Plan：想法可以被 Aide 拆成任务草稿。
- Do：用户确认后，任务进入任务列表并执行。
- Check：完成任务和实际行为记录一起进入复盘分析。
- Act：Aide 把复盘结果反馈为新的想法输入、任务建议或习惯调整。

### 四象限分类

任务需要支持重要/紧急四象限：

- 重要且紧急
- 重要但不紧急
- 不重要但紧急
- 不重要且不紧急

Aide 未来应该根据任务的重要程度、紧急程度、截止日期、定期规则和用户状态，自动帮助安排任务。

优先级不应该主要依赖用户手动输入。Aide 应根据重要度和紧急度自动评估优先级。

初始规则：

- 重要但不紧急：超高优先级。长期成长、健康、学习、系统建设等事情通常属于这一类，应该主动安排。
- 重要且紧急：高优先级。必须处理，但如果长期都集中在这一类，说明系统过于被动。
- 不重要但紧急：中优先级。需要控制投入，避免被紧急感牵着走。
- 不重要且不紧急：低优先级。尽量减少或不做。

产品倾向：鼓励用户优先处理“重要但不紧急”的事情，因为这类事情最容易被拖延，但最能改善长期状态。

### 任务页面体验

任务页面应该降低输入阻力。

需求：

- 页面切换顺序优先为：今日、任务、财务、想法、行为。
- 手机端导航放在底部，方便点击。
- 电脑端导航放在顶部或侧边，方便点击并减少操作。
- 任务添加入口应改为“任务/会议添加”，默认只显示一个内容输入框。
- 会议添加归并到任务/会议添加入口中，通过详细设置里的添加类型切换。添加类型明确分为 4 种：任务、会议、定期任务、定期会议。
- 必填输入项旁边需要显示红色 `*`，让用户一眼知道必须填什么。
- 选择不同添加类型后，只显示该类型需要输入的字段，避免同时出现“类型=定期任务”和“这是定期任务”这种重复输入。
- 任务/会议添加的分类默认选“公司”，因为当前主要会议和工作任务更容易从公司上下文进入；用户可以手动改成个人。
- 定期会议设置应参考 Outlook 的心智模型：先设置“重复模式”（每天、每周、每月、每年、星期、月/日），再设置“重复范围”（从会议开始时间起，以及结束条件/备注）。
- 定期会议只有选择“定期会议”类型后才显示规则设置。必填项包括开始日期、开始时刻、结束时长（开始 + 30/60/90/120 分钟），以及重复规则。
- 定期会议的重复规则：
  - 每日：间隔 X 日，默认 1 日；选择开始日期和结束日期。
  - 每周：间隔 X 周，默认 1 周；选择开始日期和结束日期；用周一到周日 checkbox 选择重复日。
  - 每月：间隔 X 月，默认 1 月；支持指定 X 日，或第 X 个星期 X。
  - 每年：支持指定 MM-DD，或 X 月的第 X 个星期 X。
- 定期任务在现有基础上保留国历/农历选项，并采用和定期会议一致的重复规则：排除日本祝日/中国节假日、每日/每周/每月/每年、间隔、期间、星期 checkbox、指定日或第 X 个星期 X。
- “任务”默认就是非定期任务，“会议”默认就是非定期会议，不显示定期规则；只有选择“定期任务”或“定期会议”时才显示重复周期。
- 定期任务里的“可执行日期”对应重复规则的开始日期，“截止日期”对应重复规则的结束日期。
- 定期规则输入要按重复周期逐步显示：每天只显示间隔和期间；每周才显示星期 checkbox；每月/每年才显示按日期或第 X 个星期 X；每年才需要输入月份。
- 当前原型先把详细定期规则以结构化 JSON 保存，兼容旧的频率、月、日、星期字段；后续需要补一个规则计算引擎，完整支持间隔、排除项和第 X 个星期 X。
- 用户选中或展开后，才显示其他输入项，比如描述、分类、重要度、截止日期和定期规则设置。
- 非定期任务应区分“可执行日期”和“截止日期”：可执行日期表示最早什么时候能开始做，截止日期表示最晚什么时候要完成。
- 对“现在知道，但当天才能执行”的任务，应默认不进入今日 To-Do，而是进入今后 To-Do；到了可执行日期当天再进入今日 To-Do。
- 如果可执行日期早于截止日期，Aide 可以在可执行日期进入今日 To-Do，在接近截止日期时提高紧急度。
- 新增任务默认就是 To-Do，不需要选择 todo/not-to-do 类型。
- 定期任务通过“添加类型=定期任务”进入，不需要额外开关、高级规则、高级输入或“提前几天变紧急”字段。
- 任务优先级由 Aide 根据重要度和紧急度自动评估，不需要用户手动输入。
- 新增任务的紧急度默认由 Aide 根据截止日期和定期属性评估；任务一览编辑里可以保留紧急度修正入口，方便用户纠正 Aide 的判断。
- 任务一览中只显示 todo 类任务。
- 任务一览应该显示全部未完成 To-Do，用筛选和分组降低浏览压力；今日简报才严格显示今天该做的任务。
- 非定期任务和定期任务都可以长期留在任务一览中，方便查看完整计划。
- 今日简报才负责过滤出今天新建、截止日已到、已经逾期或进入提醒窗口的任务。
- 定期任务进入提醒窗口后，Aide 应在展示上把它提升为超高优先级。
- 定期任务在任务一览中应按每日、每周、每月、每年分组显示，让长期任务更容易扫描。
- 会议也归并到任务一览中，作为“会议”项显示时间、来源、地点和备注。非定期会议使用“参加”按钮，定期会议使用“结束定期”按钮。
- 会议根据自己的个人/公司和非定期/定期属性参与任务一览筛选。
- 会议不单独成组，应该归并到非定期、每日定期、每周定期、每月定期、每年定期等分组里。
- 任务一览里的定期任务和定期会议需要支持“结束定期”。结束定期表示归档整个任务或会议本体，不是记录今天的一次完成/参加。
- 任务或会议本体结束定期后，不再显示在任务一览的定期分组里，也不再进入今日 To-Do。
- 任务一览中的任务可以展开编辑，至少支持修改标题、描述、重要度、紧急度和截止日期。
- 优先级高的任务排在前面，同等情况下优先显示重要度、紧急度更高或截止日期更近的任务。
- 如果截止日期已经接近，Aide 应自动提高该任务的紧急度，避免临近任务被忽略。
- Not-to-do 不直接混在任务一览中，而是通过打开/折叠的方式查看。
- 任务一览需要显示任务的重要度和紧急度。
- 任务一览需要支持用多选框筛选：个人、公司、非定期、定期。
- 添加任务后，任务应该立即显示在任务一览中。

### 已完成任务

完成的任务和已参加的非定期会议应该可以在“已完成/已参加”里查看，但不要默认展示全部历史，避免页面变成档案库。已参加的非定期会议不应继续显示在任务一览的“非定期”分组里。

查看方式：

- 最近 1 天
- 最近 3 天
- 最近 1 周
- 最近 1 个月
- 最近 X 年
- 按关键词搜索

这个功能既可以用于回顾执行情况，也可以帮助用户找回以前做过的事情。

当前原型已支持在浏览器 UI 中按时间范围和关键词查看已完成任务，以及从行为记录中归档的已参加非定期会议。

### 大任务和 WBS

大的任务应该可以做 WBS 拆分。

未来 UI 可以把大任务可视化为树状结构、清单、时间线或其他适合执行的结构，而不是只显示成一个模糊的大 todo。

### 和 Not-To-Do 的冲突提醒

如果 task 违背了 not-to-do，Aide 应该给出警告。

例如，一个任务如果违反法律边界、道德良心、公司规则、家庭原则或身体健康原则，系统应该在计划或执行前提醒用户。

## 行为记录

除了 task，Aide 还应该支持记录用户实际做过的事情。

这些事情不一定提前出现在 task list 里，也不一定需要计划，但它们对长期分析很有价值。

示例：

- X 点上了一次厕所。
- X 日去了一趟 X 地。
- X 点吃了某种食物。
- X 点睡觉或起床。
- X 日运动了一次。
- X 日出现了某种身体不舒服。
- X 日做了某个临时决定或临时行动。

这些记录更接近 life log 或 activity log，不应该强行当成 todo。

未来用途：

- 推测健康程度。
- 分析行为趋势。
- 发现不良习惯。
- 发现和强化良好习惯。
- 辅助生成每日、每周、每月复盘。
- 和 task、not-to-do、财务记录、AI memory 产生关联。
- 和已完成任务一起生成 PDCA / STOW 分析，为新的想法和任务拆分提供数据。

行为记录需要尽量低摩擦，最好可以快速输入一条自然语言记录，之后再由系统自动分类、打标签或生成结构化数据。

行为记录输入默认只显示必须填写的内容，例如“做了什么”。分类、时间、备注等字段应该收进详细设置里，用户需要时再展开。

## Not-To-Do

Not-to-do 应该是一等公民，不只是普通备注。它代表用户明确不想做、不能做、或不应该做的事情。

初始分类：

- 违法的事情不做。
- 违背道德良心的事情不做。
- 违背公司规则的事情不做。
- 对家人不好的事情不做。
- 对自己身体不好的事情不做。

用户应该可以随时在这些分类下添加新的 not-to-do。

这些规则未来应该可以用于检查 task，并在发现冲突时给出提醒。

Not-to-do 不应该有“完成”概念，因为它代表持续有效的边界，而不是一次性任务。

Not-to-do 的输入输出应该保持简单：

- 输入：一句边界内容，加一个分组。
- 分组：法律、道德、公司、家庭、健康。
- 输出：按分组折叠/展开查看，不混入普通任务一览。
- 暂时不提供高级输入、Markdown 或 YAML。复杂原则、例外条件和外部规则以后可以由 Aide 逐步结构化。

## 财务

财务功能的长期方向可以接近 Money Forward。

想法包括：

- 通过各个银行的 API 获取或同步数据。
- 可视化当前财务状态。
- 记录收入、支出、账户、资产等信息。
- 记录财富积累目标。
- 计算距离实现财富自由还差多少钱。

这个方向会涉及隐私、安全、银行 API、数据导入和可视化，属于比较大的功能模块。

### 财务页面体验

当前阶段财务页面不需要很细的流水记录。

更重要的是作为“财富自由仪表盘”，快速回答：

- 现在有多少真正参与财富自由计算的 FI 资产。
- 每月家庭固定支出项目合计是多少。
- 每月被动收入是多少。
- 被动收入覆盖率是多少。
- 按安全提取率反推，FI 目标本金是多少钱。
- 距离实现财富自由还差多少钱。
- 按当前每月新增投资，还需要多久。

流水、分类、银行 API 和复杂可视化可以放到后面。当前 UI 应优先展示财务自由相关的关键数字。

财富自由定义：

- 核心公式：被动收入 >= 生活总支出。
- 本质：不再为了生活开销被迫出售时间，获得时间自由和时间支配权。
- 真正参与 FI 计算的资产应该是能产生现金流、投资收益或未来可变现投资收益的资产。
- 自住房、车、难变现物品等可以记录为非 FI 资产，但默认不计入 FI 目标计算。

当前计算规则：

现金流法：

- 被动收入覆盖率 = 每月被动收入 / 每月生活支出。
- 每月生活支出由固定支出项目加总，例如房租、游泳、饮食、水电网等。
- 每月现金流差额 = 每月被动收入 - 每月固定支出，可以是负数。
- 如果每月被动收入已经覆盖每月生活支出，FI 差距可以视为 0。

本金法：

- FI 目标本金 = 年固定支出 / 安全提取率。
- FI 差距 = FI 目标本金 - 当前 FI 资产。
- 每月新增投资不改变 FI 目标本金或 FI 差距本身，但用于估算补齐 FI 差距还需要多久。
- 安全提取率需要可调，默认使用 3.5%，也可以按 3%、4% 等假设查看。

当前原型已将财务页改为 FI 仪表盘，支持 FI 资产、非 FI 资产、固定支出项目增删、每月被动收入、每月新增投资和安全提取率。UI 应将现金流法和本金法分组展示，并用短文字说明每个数字代表什么。

### 多币种资产和汇率

资产输入需要支持多种资产类型：

- 日元
- 人民币
- 美金
- 虚拟货币，例如 BTC、ETH

显示需求：

- 最终结果可以用日元或人民币展示。
- 财务页面需要有日元/人民币切换按钮。
- 汇率根据当天汇率自动计算。
- 资产金额必须标明单位。
- FI资产、FI目标本金、还差金额、每月支出、每月被动收入、现金流差额、非FI资产都需要同时显示原始大数字和括号里的小字概算。
- 小字概算用更容易理解的单位，例如“约 X 万”、“约 X 千万”、“约 X 亿”。
- 括号里的金额只放金额概算，不混入提取率、状态说明等其他文本；其他说明放在单独小字里。
- 距离财务自由还差的时间，需要根据每月新增投资自动换算成“X年X个月X天”。
- 每月新增投资额需要在本金法里显示，并说明它只影响“还需要多久”，不改变 FI 目标本金或本金缺口本身。
- 金额输入框应尽量像银行 App 一样，只接受数字输入，并自动添加千分位逗号。

当前阶段可以先做本地浏览器保存和前端换算。后续再考虑把财务概况持久化到后端数据库。

## Thoughts

用户已经有 Joplin 笔记本，可以随时记录想法。

所以 Aide 里的 thoughts 功能目前可以不是核心优先级。未来可以考虑：

- 保留一个轻量 capture 功能。
- 以后和 Joplin 集成。
- 从 Joplin 中导入部分笔记进入 Aide memory。
- 只把需要 follow-up、转成任务、或进入 AI memory 的内容放进 Aide。

想法页面应作为独立页面，不和任务、行为记录混在一起。

想法输入也应该低摩擦。默认只显示必须填写的内容，例如“内容”。标签等字段放进详细设置里。

想法未来应该接入 AI，但重要决策必须经过用户确认：

- 用户提交想法后，Aide 可以把想法拆成几个任务草稿。
- Aide 先展示拆分结果，说明每个任务的标题、上下文、重要度和紧急度。
- 只有用户明确同意后，任务才会真正加入任务列表。
- 拆分后的任务继续进入任务一览、完成状态和后续 PDCA。

当前原型先用本地规则生成任务草稿，不调用外部 AI。后续接入 AI provider 后，应保留“先建议、再确认、再写入”的交互。

行为记录也应该接入 AI：

- Aide 汇总已完成任务和用户提交的行为记录。
- Aide 分析 STOW：Strengths、Threats、Opportunities、Weaknesses。
- Aide 同时给出 PDCA 草稿，帮助用户理解计划、执行、检查和调整。
- 行为分析结果应该反过来为想法页面提供数据，例如提示哪些行为值得变成任务、哪些习惯值得强化、哪些风险需要避免。

## 公网访问和认证

用户计划通过 Cloudflare Tunnel 把 Aide 放到公网访问。

认证需求：

- 安全性要比较高。
- 操作要简单。
- 尽量不需要记住密码。
- 因为用户主要只有自己，所以访问控制可以围绕自己的邮箱、手机、电脑和可信设备设计。

倾向方案：

- 使用 Cloudflare Access 保护 Aide。
- 使用邮箱一次性验证码、Google 登录、Passkey 或设备策略，而不是先自己实现用户名密码。
- 只允许用户自己的邮箱或可信设备访问。
- 后端未来可以校验 Cloudflare Access 注入的身份信息或 JWT，作为额外保护。

## 当前优先级倾向

根据这些想法，当前产品优先级可以倾向于：

1. 强化 task 系统：定期任务、每日简报、PDCA、四象限、WBS。
2. 增加行为记录：记录实际发生过的事情，用于健康、习惯和趋势分析。
3. 优化移动端和桌面端 UI：减少操作，首页聚焦今日 To-Do、今后 To-Do 和简短今日简报。
4. 强化 not-to-do：分类规则、随时添加、task 冲突提醒。
5. 扩展财务：先展示 FI 资产、每月支出、被动收入覆盖率、FI 目标本金和差距，再考虑复杂流水。
6. 做好公网访问认证：优先使用 Cloudflare Access，不急着自研密码系统。
7. 降低 thoughts 优先级：先作为独立可选页面，未来考虑 Joplin 集成或 AI memory 输入源。
